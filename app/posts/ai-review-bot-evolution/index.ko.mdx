## 왜 PR 리뷰봇을 만들게 되었을까?

### 개발 속도가 리뷰를 앞지르는 순간

AI 코딩 도구의 도입 이후, 우리 팀의 개발 속도는 체감할 수 있을 정도로 빨라졌어요. 문제는 그 속도를 리뷰가 따라가지 못한다는 거예요.

PR이 올라오는 속도는 빨라졌는데, 리뷰어는 여전히 같은 인원이죠. 자연스럽게 리뷰 대기열이 쌓이고, **리뷰가 배포의 병목**이 되는 현상이 반복됐어요. "코드는 30분 만에 짰는데 리뷰 대기가 하루"같은 상황이 일상이 된 거예요.

### 리뷰어 개인에게 집중되는 부담

리뷰해야 할 PR이 늘어나면서 인간 리뷰어 개인이 감당해야 하는 부담도 커졌어요. 내가 리뷰를 늦게 하면 동료의 배포가 밀리고, 테스트가 늦어지죠. 이 압박감은 리뷰의 질을 떨어뜨리거나, 반대로 리뷰어의 피로를 가중시켰어요.

### TBD 도입과의 충돌

우리 FE 팀은 이 시기에 Trunk-Based Development(TBD)를 도입하고 있었어요. TBD는 작은 단위의 PR을 빠르게 머지하는 것이 핵심인데, 리뷰 병목은 이 흐름과 정면으로 충돌했죠. 짧은 수명의 브랜치를 빠르게 머지해야 하는데 리뷰에서 막히면, TBD의 장점이 사라지는 거예요.

<strong>"기본적인 코드 품질 검증을 자동화해서 인간 리뷰어의 부담을 줄일 수 없을까?"</strong> — 이것이 리뷰봇을 만들게 된 출발점이었어요.

## 리뷰봇 개선 타임라인

3개월간 리뷰봇을 운영하면서 7번의 큰 개선을 거쳤어요. (작은 수정들은 정말 많았지만요)
각 Phase에서 어떤 문제를 만났고, 왜 그런 선택을 했는지 공유해볼게요.

<Timeline>
  <TimelineItem title="Phase 1 — 초기 구축" summary="Claude 기반 도입, triage로 비용 제어">

  처음에는 Claude 기반으로 모든 변경 파일을 리뷰하게 했어요.

  결과는 **비용 폭발**. PR 하나에 수십 개 파일이 변경되면, 전부 리뷰를 돌리는 건 API 비용이 감당이 안 됐죠.

  여기서 첫 번째 핵심 설계가 나왔어요: **triage(사전 분류)**. 가벼운 모델(Haiku)로 "이 파일이 리뷰가 필요한가?"를 간단한 근거를 기준으로 먼저 판단하고, 필요한 파일만 정밀 리뷰하는 2단계 구조를 만들었어요.

  > "전부 리뷰"가 아니라 <strong>"무엇을 리뷰할지 판단"하는 것부터 시작</strong>해야 해요. triage는 비용 절감뿐 아니라 노이즈 감소에도 결정적이었어요.

  </TimelineItem>

  <TimelineItem title="Phase 2 — 노이즈 제거" summary="엄청난 노이즈">

  리뷰봇이 돌아가고 PR에 코멘트가 많이 쌓이게 되면서 실질적으로 팀원들이 리뷰를 할 때 도움이 되는 게 아니라 오히려 악영향을 끼치게 되었어요

  useEffect dependency 경고, TypeScript 타입 이슈 같은 **프레임워크나 컴파일러가 이미 잡아주는 것들을 봇이 중복 지적**하니 PR 작성자 입장에서는 굳이 리뷰하지 않아도 되는 걸 리뷰하고 그에 대해 코멘트를 달거나 resolve comment를 해야 하다 보니 하나의 PR에 의미 없는 코멘트만 20~30개가 달릴 때도 있었어요

  또 하나의 문제는 **rebase만 해도 리뷰가 다시 달리는 것**. 코드 변경 없이 rebase만 했는데 동일한 코멘트가 또 달리면 신뢰가 떨어져요.

  - 컴파일러/린터가 커버하는 영역의 리뷰를 모두 억제
  - rebase-only 변경을 감지해서 리뷰 스킵 (git tree hash 비교)

  > 봇의 가치는 "많이 지적하는 것"이 아니라 <strong>"사람만이 잡을 수 있는 것을 잡는 것"</strong>. 도구가 이미 커버하는 영역을 침범하면 봇 전체의 신뢰가 무너져요.

  신뢰가 무너진 상태로 계속 진행되면 팀원들은 자연스럽게 리뷰봇의 리뷰를 노이즈로 인식하게 되어 좋은 리뷰를 달아도 넘어갈 확률이 높아졌을 거예요

  </TimelineItem>

  <TimelineItem title="Phase 3 — 아키텍처 전환" summary="모놀리식에서 파이프라인으로 & 리뷰봇을 위한 정보 제공">

  이 Phase에서는 두 가지 큰 변경이 있었어요.

  **첫째, 모놀리스에서 파이프라인으로.** 리뷰봇 로직이 500줄짜리 단일 YAML에 몰려 있다 보니, 팀원들이 기여하고 싶어도 어디를 고쳐야 할지 감이 안 잡혔어요. 이걸 pre\_filter → triage → review → validator **4개의 독립 스크립트**로 분리했어요. 각 단계가 독립적이라 특정 단계만 수정하거나 디버깅할 수 있게 됐죠.

  **둘째, 컨텍스트 주입.** diff만 보고 PR의 의도를 오해해서 엉뚱한 지적을 하는 문제가 있었어요. **PR 제목, 설명, 커밋 메시지를 리뷰 프롬프트에 함께 주입**하기 시작했어요.

  > 리뷰봇은 file diff만 봐서는 안 돼요. "왜 이 변경을 했는지"를 모르면 의도와 동떨어진 지적을 하게 되고, 이 부분이 노이즈의 큰 원인이라고 생각했어요.

  </TimelineItem>

  <TimelineItem title="Phase 4 — 규칙의 역설" summary="추측 금지를 너무 강하게 걸면 리뷰가 사라진다">

  Phase 3에서 구조를 개선했지만 여전히 노이즈가 발생했어요. 반복적으로 노이즈를 일으키는 리뷰 패턴들을 특정해서, "이런 리뷰는 하지 마"라는 금지 규칙을 프롬프트에 추가했죠.

  그런데 금지 규칙을 강하게 적용하니, **리뷰 자체가 거의 나오지 않는** 역효과가 발생했어요. 봇이 지나치게 보수적으로 변한 거예요.

  이 문제를 해결하기 위해 **생성과 검증을 분리**했어요:

  - **code\_review\_guide** — 톤, 심각도 기준, 금지 규칙을 명문화한 가이드 문서
  - **validator 스크립트** — 리뷰 생성 후 별도 모델이 "이 코멘트가 가이드를 지키는가?"를 검증

  즉, **생성(review)과 검증(validator)의 2단계 구조**가 이 Phase에서 처음 등장했어요.

  > 프롬프트에 금지 규칙을 쌓는 것만으로는 품질 통제가 어려워요. 생성과 검증을 분리하면 "이건 하지 마"보다 훨씬 안정적으로 품질을 통제할 수 있어요.

  </TimelineItem>

  <TimelineItem title="Phase 5 — 2-Pass Self-Critique" summary="같은 모델, 다른 관점 — 리뷰어와 작성자를 동시에 시뮬레이션">

  Phase 4에서 도입한 validator를 본격적으로 개선한 단계예요. 우리가 **2-pass review**라고 부르는 구조의 완성이에요.

  **1st pass — 리뷰어 역할.** 리뷰 모델이 PR diff와 컨텍스트를 보고 코멘트를 생성해요. 여기까지는 일반적인 AI 리뷰와 같죠.

  **2nd pass — PR 작성자 역할.** 같은 수준의 모델이 이번엔 역할을 바꿔요. "너는 이제 이 PR의 작성자야. 이 코멘트들을 받았을 때, 도움이 된다고 느끼겠어?" 이 관점에서 1st pass의 코멘트를 하나씩 평가하고, 가치 없는 코멘트를 걸러내요.

  여기서 핵심은 **few-shot 예시**예요. 팀원들이 실제로 봇 코멘트에 남긴 반응 데이터(좋아요/싫어요)를 수집해서, "이런 코멘트는 좋은 반응을 받았다 / 이런 코멘트는 나쁜 반응을 받았다"를 validator에 예시로 넘겨요. 모델이 추상적인 규칙이 아니라 **팀의 실제 선호도를 기준으로** 판단하게 되는 거죠.

  추가로 validator는 지시형 어투를 질문형으로 변환하고("~하세요" → "~한 이유가 있을까요?"), 코멘트를 최대 7개로 제한하고, 유사한 코멘트는 가장 중요한 1개만 남기는 등의 후처리도 수행해요.

  > 2-pass 구조는 단순히 노이즈를 줄이는 것을 넘어, **리뷰의 관점을 전환**시켜요. "코드가 규칙에 맞는가"에서 "이 피드백이 작성자에게 가치가 있는가"로.

  </TimelineItem>

  <TimelineItem title="Phase 6 — 컨텍스트의 완성" summary="diff만 보지 말고, 설계 문서를 읽어라">

  AI 하네스 엔지니어링으로 개발하다 보면 필연적으로 spec, plan, design 같은 설계 문서를 작성하게 돼요. 
  보통 이런 문서는 개발이 끝나면 휘발되기 마련이고, 누군가는 PR을 작성할 때 의도적으로 제거할 수도 있죠. 
  이번에는 오히려 리뷰봇에게 컨텍스트를 전달할 수 있는 좋은 자산이라고 판단했어요.

  - PR 브랜치명/설명에서 관련 spec/plan 문서를 **자동 탐지**
  - review와 validator 양쪽에 본문을 컨텍스트로 전달
  - **plan과 구현 사이에 차이가 있다면 의도를 물어봐라**로 프롬프트 변경

  > 개인적으로 가장 큰 임팩트를 준 변화였어요. 설계 의도를 아는 봇은 "왜 이렇게 했지?"라는 질문 대신 **plan에서는 A라고 했는데 구현은 B인데, 의도한 건가요?**라는 질문을 해요.

  </TimelineItem>

  <TimelineItem title="Phase 7 — 자율 학습 루프" summary="사람의 반응을 학습 데이터로 자동 전환">

  지금까지 모든 개선은 사람이 직접 프롬프트를 수정하는 방식이었어요. 하지만 개발자들은 이미 매일 PR 코멘트에 반응을 남기고 있죠 — 좋아요/싫어요, resolve, 답글, dismiss. 이 시그널이야말로 가장 솔직한 피드백 데이터라고 생각했어요.

  매주 한 번 scheduled action이 돌면서 지난 1주일간의 PR 코멘트 반응을 자동 수집해요. 좋은 반응을 받은 코멘트와 나쁜 반응을 받은 코멘트를 분류하고, 이 데이터를 기반으로 리뷰봇 개선안을 생성해요. 단순 리포트가 아니라 실제 적용 가능하다고 판단되면 개선안을 담은 PR까지 자동으로 올리는 프로세스예요.

  - 1주일치 봇 코멘트의 팀 반응(좋아요/싫어요, resolve, dismiss 등)을 **자동 수집**
  - 좋은 지적 / 나쁜 지적을 분류해서 **few-shot 예시로 축적**
  - 개선이 필요하다고 판단되면 **프롬프트 수정 PR을 자동 생성**

  사람이 프롬프트를 직접 수정하지 않아도 봇이 팀의 선호도에 맞춰 점진적으로 개선되는 구조예요.

  > 리뷰봇의 최종 형태는 "잘 만든 프롬프트"가 아니라 **팀의 피드백을 자동으로 반영하는 시스템**이라고 생각해요.

  </TimelineItem>
</Timeline>

## 도입 전 vs 도입 후 — 리뷰 문화는 어떻게 바뀌었을까?

### 인간 리뷰어의 역할 변화

리뷰봇 도입 전에는 인간 리뷰어가 모든 것을 봐야 했어요. 타입 실수, import 순서, 네이밍, 비즈니스 로직, 설계 의도까지. 봇 도입 후에는 **기본적인 코드 품질과 컨텍스트 기반 검증은 봇이 처리**하고, 인간 리뷰어는 비즈니스 로직과 설계 판단에 집중할 수 있게 됐어요.

### Bypass 문화의 등장

흥미로운 변화 중 하나는 **bypass(빠른 승인) 방식이 강하게 사용되기 시작한 것**이에요. 봇이 기본 검증을 이미 끝낸 PR에 대해, 인간 리뷰어가 빠르게 확인하고 승인하는 패턴이 자연스럽게 자리잡았죠. 봇이 "놓친 게 없는지" 한 번 확인해주니까, 인간 리뷰어 입장에서는 전체를 꼼꼼히 볼 필요 없이 봇이 플래그한 부분과 비즈니스 로직에만 집중하면 되는 거예요.

### 리뷰 대기 시간 감소

TBD 흐름과도 잘 맞물렸어요. 봇 리뷰는 PR이 올라오면 즉시 실행되므로 시간대와 무관하죠. 인간 리뷰어가 오프라인이어도 기본적인 피드백은 바로 받을 수 있고, 리뷰어가 돌아왔을 때 이미 봇이 기본 검증을 끝낸 상태이므로 리뷰 시간이 크게 줄었어요.

## 마치며

7번의 큰 여정을 거치면서 가장 크게 배운 것은, <strong>AI 리뷰봇은 "코드를 잘 읽는 AI"가 아니라 "팀 문화에 맞는 피드백 시스템"</strong>이라는 거예요.

단순히 프롬프트나 yml 파일을 수정하는 게 아니라 중요한 것은 <strong>"어떤 리뷰가 팀에게 가치가 있는가"</strong>를 정의하는 것이었고, 그 정의는 팀의 반응 데이터에서 나왔어요. 결국 좋은 리뷰봇은 좋은 리뷰 문화의 반영이라고 생각해요.

이 과정에서 우리는 PR 작성자와 리뷰어, 양쪽의 관점을 모두 고민해야 했어요.

PR 작성자 입장에서, 코드 스타일이나 반복적인 UI 지적은 우리가 리뷰어에게 기대하는 리뷰가 아니었어요.
작성자가 원하는 건 "이 설계 판단이 맞는지", "이 변경이 다른 기능에 영향을 줄 수 있는지"에 대한 팀의 의견이었죠.

리뷰어 입장에서도 마찬가지예요. 엄청난 양의 UI 코드, 뭐부터 봐야 할지 모르는 PR context 를 봤을 때 "도대체 뭘 봐야 하지?"라는 생각이 드는 순간, 정작 중요한 리뷰를 놓치게 돼요.

양쪽 모두가 진짜 논의가 필요한 부분에 집중할 수 있는 리뷰 — 그것이 우리가 리뷰봇을 통해 만들고 싶었던 리뷰 문화예요.
